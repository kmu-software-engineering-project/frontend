# QA 검증 리포트
검증 일시: 2026-06-06
검증 범위: S4-7 — 무한 스크롤 `isEnd` 경계 처리 수정

## 변경 사항
`src/app/reviews/page.tsx` IntersectionObserver 생성 조건 수정
- 기존: `if (!sentinel || books.length === 0) return` + 콜백 내 `hasMore` 체크
- 변경: `if (!sentinel || books.length === 0 || !hasMore) return` + 콜백에서 `hasMore` 제거

**수정 의도**: `hasMore === false` 전환 시 observer를 아예 생성하지 않아
경계 케이스(마지막 페이지 직후)에서 중복 fetch가 발생하지 않도록 차단.
dependency array에 `hasMore`는 유지 → `false` 전환 즉시 effect 재실행 → guard에서 `return`.

- ✅ 통과: 9항목

### B. 수정 내용 정합성

**B-1. guard 조건 (`!hasMore` 추가)**
결과: 통과

`reviews/page.tsx:234` — `if (!sentinel || books.length === 0 || !hasMore) return`.

**B-2. 콜백 내 `hasMore` 체크 제거**
결과: 통과

`reviews/page.tsx:238` — `if (entry.isIntersecting && !isLoading && !isLoadingMore)`.
`hasMore` 조건이 effect 진입부에서 이미 보장되므로 콜백 내 중복 체크 제거.

**B-3. dependency array `hasMore` 유지**
결과: 통과

`reviews/page.tsx:247` — `[books.length, hasMore, isLoading, isLoadingMore, loadBooks, nextPage]`.
`hasMore`가 `false`로 전환되면 effect 재실행 → 즉시 `return` → observer 미생성.
이 의존성이 없으면 `hasMore` 변경을 감지하지 못해 이전 observer가 남아있게 됨.

### C. hasMore 로직 정합성

**C-1. setHasMore 업데이트 로직**
결과: 통과

`reviews/page.tsx:208` — `setHasMore(!data.isEnd && (data.books?.length ?? 0) > 0)`.
`isEnd === true` 또는 빈 배열 응답 시 `hasMore = false` 처리 정상.

**C-2. setHasMore(true) 초기화**
결과: 통과

`reviews/page.tsx:227` — `setHasMore(true)` — 검색어 변경(submittedQuery) 시 loadBooks 의존성 경유하여 초기화.

**C-3. hasMore=false 시 observer 미생성 논리**
결과: 통과

guard 조건에 의해 `hasMore === false && books.length > 0` 케이스에서 observer가 생성되지 않음.
이전 구조와 달리 중복 fetch 발생 경로 완전 차단.

### D. 관련 UI 요소

**D-1. sentinelRef 렌더링 배치**
결과: 통과

`reviews/page.tsx:490` — `<div ref={sentinelRef} className="h-12" aria-hidden="true" />`.
책 목록 하단에 배치, IntersectionObserver 트리거 지점으로 적절.

**D-2. 종료 메시지 조건 분기**
결과: 통과

`reviews/page.tsx:493` — `!hasMore && books.length > 0` 조건으로 "모든 검색 결과를 불러왔습니다." 메시지 표시.
`hasMore === false` 전환 후 정상 표시.
